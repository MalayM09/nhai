# Project Blueprint: Edge-AI Offline Biometric Authentication for Hackathon 7.0

### 1. Model Details
* **Facial Recognition:** **MobileFaceNet**. Chosen for its extreme efficiency using depthwise separable convolutions. It outputs a 512-dimensional embedding vector representing the facial identity.
* **Passive Liveness (Anti-Spoofing):** **ShuffleNetV2** (0.5x scale) or **MiniFASNet**. These architectures utilize channel shuffling to maintain information flow while drastically reducing computational complexity, making them ideal for identifying print and screen artifacts.
* **Face Detection & Landmarking:** **MediaPipe BlazeFace / FaceMesh** (highly compressed TFLite versions) to extract the bounding box and 2D landmarks required for alignment and heuristics.

### 2. Heuristic Details
Heuristics form the deterministic, zero-memory Gate 1 of the liveness pipeline.
* **Eye Aspect Ratio (EAR):** Calculates the ratio of distances between vertical and horizontal eye landmarks. An EAR < 0.2 across 3 consecutive frames constitutes a blink.
* **Mouth Aspect Ratio (MAR):** Calculates the vertical opening of the lips. MAR > 0.5 constitutes an open mouth/smile.
* **Euler Angles via PnP (Perspective-n-Point):** Maps 2D landmarks to a generic 3D face model to calculate head pose (Yaw, Pitch, Roll). A Yaw shift of +/- 25 degrees validates a head turn.

### 3. How to Use Heuristics with the Model
You must implement a **Cascading Gating Architecture**. 
Do not run the neural networks simultaneously. The React Native app must prompt the user with a randomized challenge (e.g., "Look left"). The C++ frame processor calculates the PnP Yaw heuristic. *Only* if the Yaw threshold is crossed does the pipeline capture that specific frame, crop the face, and pass it to ShuffleNet (Gate 2) and MobileFaceNet (Gate 3). If Gate 1 fails, the frame is dropped, saving battery and compute.

### 4. Model Sizes and Constraints
The strict cap is ~20 MB for the entire application bundle. 
* React Native Base App + Libraries: ~8-10 MB.
* MobileFaceNet (INT8 Quantized): ~2-3 MB.
* ShuffleNetV2 (INT8 Quantized): ~1.5 MB.
* BlazeFace (Detection): ~0.5 MB.
* **Total App Size:** ~13-15 MB (Safely under the 20 MB cap).
* **Inference Constraint:** Must process < 1 second. By quantizing to INT8 and using the cascading gates, end-to-end inference on a 3GB RAM device (Android 8 / iOS 12) will take roughly 300-450ms.

### 5. Dataset Used (General)
You will not use raw images in the final app. You will only use datasets to train/fine-tune your `.tflite` models before packaging them. 

### 6. Pretraining Dataset Used
* **MS1MV3** or **Glint360K:** Use weights pre-trained on these massive datasets. Do not train MobileFaceNet from scratch on your local machine; leverage open-source pre-trained weights to ensure high baseline generalization.

### 7. Fine-Tuning Dataset Used
* **Indian Movie Face Database (IMFDB):** To meet the requirement of recognizing diverse Indian demographics, fine-tune the pre-trained MobileFaceNet on IMFDB.
* **Augmentation Strategy:** To handle varying outdoor lighting and harsh sunlight, aggressively augment your fine-tuning data with extreme brightness shifts (gamma correction), synthetic shadows, and Gaussian blur.

### 8. Heuristic Modeling and Dataset Used
Heuristics **do not require training datasets**. They rely entirely on pre-trained landmark detectors (MediaPipe) and hardcoded mathematical geometry (like the standard 3D points of a human face). You only need to calibrate the thresholds (e.g., EAR < 0.2) based on manual testing with your team.

### 9. Integrating the Vision Model with the Heuristic Model
Integration happens at the native bridge layer. The camera outputs a raw buffer. 
1. Run BlazeFace (detect face).
2. Calculate Heuristics on landmarks (Gate 1).
3. If Gate 1 passes, execute an Affine Transformation (rotation/scaling) using the eye landmarks to align the face perfectly horizontally.
4. Feed the aligned tensor to the AI models. 

### 10. Loss Functions Used & Validation Strategies
* **Loss Function:** **ArcFace (Additive Angular Margin Loss)**. Standard Softmax does not create enough margin between different faces. ArcFace projects embeddings onto a hypersphere and forces a distinct angular margin between different identities, crucial for > 95% accuracy.
* **Validation:** Use K-Fold cross-validation on your fine-tuned demographic dataset.

### 11. Metric Used for the Loss Functions
To validate the model (and set the app's threshold), plot the **ROC (Receiver Operating Characteristic) Curve**. Calculate the **EER (Equal Error Rate)**—the point where False Acceptance Rate (FAR) equals False Rejection Rate (FRR). This exact Euclidean or Cosine distance value at the EER becomes the hardcoded threshold in your React Native app.

### 12. Integrating the Complete Model with the Mobile App
Do not use JavaScript for inference. Convert all PyTorch/TensorFlow models to `.tflite` or `.onnx`. Bundle these files directly in the `android/app/src/main/assets` and iOS equivalent folders. Execute them natively using C++ via React Native JSI (JavaScript Interface) to prevent bridge serialization bottlenecks.

### 13. Libraries Used in React Native for Maximum Optimization
* **Camera & JSI:** `react-native-vision-camera` (The absolute gold standard for 60fps frame processing).
* **Inference:** `react-native-fast-tflite` (Uses CoreML on iOS and NNAPI on Android natively).
* **Database:** `react-native-quick-sqlite` (C++ based SQLite wrapper, exponentially faster than standard async storage for querying vectors).

### 14. How to Take Inferences and App Workflow
1. Camera frame is captured at 480p (downscale for speed).
2. Frame Processor (JSI) intercepts the frame before rendering.
3. TFLite executes detection -> heuristics -> alignment -> MobileFaceNet.
4. The output is an array of 512 floats `[0.12, -0.45, ... ]`.
5. A C++ helper function calculates the Cosine Distance between this array and the arrays in SQLite.

### 15. Complete App Usage Lifecycle
The app operates entirely offline in the field. The user opens the app, the UI prompts a liveness challenge, the user complies, and the app matches the face. A success screen appears, and an attendance log (User ID + Timestamp) is saved locally.

### 16. How to Log a New User (Enrollment)
Enrollment must happen when the device *has* an active network connection (e.g., at the regional office before heading into the field). 
1. Admin logs in.
2. Scans the new user's face under good lighting.
3. App extracts the 512-D embedding. 
4. App sends the embedding and User ID to your AWS backend.
5. The backend syncs this new identity down to all offline field devices.

### 17. Keeping Embeddings in the App
Users do nothing. The system handles it. The AWS server acts as the master ledger. Whenever a field device connects to Wi-Fi/cellular, it silently pings a standard backend (e.g., a FastAPI endpoint) to download the latest embedded vectors for the target region and update its local SQLite database. 

### 18. Storing and Using Embeddings
Embeddings are stored in the local **SQLite** database. 
* **Schema:** `CREATE TABLE users (id TEXT PRIMARY KEY, name TEXT, embedding BLOB);`
* The 512 floats are serialized into a binary BLOB or a JSON string. 
* During inference, the app loads these vectors into memory (RAM) as a 2D matrix. Matrix multiplication against the live camera vector allows matching against hundreds of users in < 10 milliseconds.

### 19. End-to-End Architecture (Minute Details)
* **Frontend (React Native):** Implement a minimalist, corporate-professional aesthetic (think clean typography, ample white space, and specific, muted color palettes). Avoid heavy animations. Use simple status cards (e.g., "Ready to Scan", "Verified").
* **Local Storage:** `react-native-quick-sqlite`.
* **Backend Sync Queue:** Implement an offline queue. Attendance logs are stored with a `synced = 0` flag. When the `NetInfo` library detects a connection, a background job pushes the records to your AWS backend via a REST API (built on FastAPI for high concurrency).
* **Purge Mechanism:** Once AWS returns a `200 OK` confirming receipt, the app runs `DELETE FROM attendance WHERE id = ?` to purge the local data.

### 20. Optimizing the App
* **Camera Resolution:** Do not capture in 4K or 1080p. 480p is mathematically sufficient for MobileFaceNet and reduces memory overhead by 75%.
* **Frame Throttling:** Do not run inference on every single frame. Throttle your frame processor to run every 3rd or 4th frame (~10 FPS). 
* **Memory Leaks:** Explicitly release memory buffers in your C++ frame processor after inference. If you leave YUV buffers hanging, the app will crash after 30 seconds.

### 21. Fallbacks
* **Lighting Fallback:** If BlazeFace fails to detect a face due to low light, trigger the device's screen brightness to 100% and paint the UI white to act as a makeshift ring light.
* **Heuristic Fallback:** If the user fails to blink (e.g., wearing sunglasses), the state machine should timeout and switch the prompt to "Turn your head left."
* **Hardware Fallback:** If the Neural Processing Unit (NPU) fails to initialize the `.tflite` model, catch the error and fallback to CPU execution using a lower-thread count to prevent thermal throttling.

### 22. Crucial Missing Elements for Hackathon Success (Additions)
* **Security (Database Extraction):** The brief demands the system be "secure". If someone steals the offline device, they can extract the SQLite database. While images aren't there, embeddings can technically be reverse-engineered. You **must** use SQLCipher to encrypt the local SQLite database using a key stored in the iOS Keychain / Android Keystore. Highlight this to the judges; it will score massive points for feasibility and security.
* **Sensor Orientation & Affine Alignment:** Mobile cameras return frames rotated 90 or 270 degrees by default depending on device orientation. If you pass a sideways face to MobileFaceNet, it will output garbage. You must utilize the device gyroscope/orientation sensors to mathematically rotate the tensor buffer before feeding it to the models.
* **The Zero-Network Edge Case (Clock Spoofing):** Users in offline zones can change the system time on their Android device to commit attendance fraud (e.g., clocking in at 9 AM when it is actually 12 PM). You must implement an offline monotonic clock check (tracking device uptime instead of wall-clock time) to detect if the user altered the OS time while disconnected from the network.
